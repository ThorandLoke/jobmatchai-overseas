# 产品安全迭代工作流 — 通用模板

> **适用场景**：任何已上线产品的功能迭代、新功能开发、问题修复
> **核心原则**：生产代码不动，新功能在隔离环境中开发，通过测试后才部署
> **模板版本**：v1.1 | 制定日期：2026-04-07 | 更新：2026-04-08
> **团队**：ThorandLoke（Loke 负责开发，Thor 负责需求和验收）

## 一、三环境原则

```
┌─────────────────────────────────────────────────┐
│  🏠 开发环境（DEVELOPMENT）                       │
│  · 本地机器，隔离于互联网                        │
│  · 直接读写代码                                  │
│  · 用于：写代码、调试、初步验证                   │
│  · 访问：localhost:xxxx                          │
└─────────────────────────────────────────────────┘
                      ↓ 测试通过后
┌─────────────────────────────────────────────────┐
│  🔬 预发布环境（STAGING）                        │
│  · 独立地址，不影响生产                          │
│  · 与生产环境配置一致（数据库、API等）            │
│  · 用于：完整功能测试、UAT（用户验收测试）         │
│  · 访问：staging.xxx.com / onrender staging 等   │
└─────────────────────────────────────────────────┘
                      ↓ Thor 书面确认
┌─────────────────────────────────────────────────┐
│  🚀 生产环境（PRODUCTION）                       │
│  · 真实用户访问的唯一地址                         │
│  · 代码只读，不直接修改                          │
│  · 任何变更必须走完整流程                         │
│  · 访问：正式域名                                │
└─────────────────────────────────────────────────┘
```

## 二、版本隔离策略

### 分支命名规范

| 分支类型 | 命名格式 | 用途 |
|---------|---------|------|
| 主线 | `main` | 当前生产可用的稳定代码 |
| 功能分支 | `feature/功能名称` | 新功能开发 |
| 修复分支 | `fix/问题描述` | Bug修复 |
| 实验分支 | `experiment/实验名` | 探索性开发，不保证合并 |
| 发布标签 | `v主.次-stable` | 记录已发布版本快照 |

### 关键规则

1. **生产代码不动** — `main` 分支只在发布前合并，不在生产环境直接改代码
2. **功能完成后合并** — `feature/` 完成测试后合并到 `main`，触发自动部署
3. **失败回退** — 合并后发现问题，打标签记录，用 `git checkout vX.X-stable` 恢复到上一个稳定版本

## 三、标准迭代流程（6步）

```
Step 1: 想法          Thor 提出想法
Step 2: 文档          创建 FEATURE-xxx.md，记录背景和设计
Step 3: 开发          在 feature/ 分支写代码，不碰 main
Step 4: 测试          本地验证 → 提交 GitHub → 自动部署 staging
Step 4.5: 代码审查    Loke 自审 + 教训文档检查
Step 5: 验收          Thor 在 staging 测试，书面确认
Step 6: 部署 & 监控   合并 main → 生产 → 观察48小时
```

### Step 1 — 想法
- Thor 用自然语言描述想要什么
- Loke 评估可行性，返回方案建议
- **无需写文档，口头讨论即可**

### Step 2 — 文档（FEATURE 文件）
**触发条件**：想法确认要做，进入开发阶段

创建文件：`[项目]/docs/FEATURE-功能名.md`，模板见本文档末尾。

文档内容必须包含：
- 背景与目标（为什么做）
- 功能设计（做什么）
- 变更记录（每次调整的原因）

### Step 3 — 开发
- 从 `main` 新建 `feature/xxx` 分支
- 所有代码在 `feature/` 分支内完成
- **禁止在 `main` 分支上直接开发**

```bash
cd [项目目录]
git checkout main
git pull origin main
git checkout -b feature/功能名
# ... 开发中 ...
git add . && git commit -m "feat: 描述"
```

### Step 4 — 测试
**本地测试**：
```bash
# 启动本地开发服务器
cd [项目目录]
[启动命令，如 uvicorn main:app --reload 或 npm run dev]
# 访问 localhost:xxxxx 验证功能
```

**提交后自动部署到 staging**（如配置了 CI/CD）：
```bash
git push origin feature/功能名
# Render / Vercel 自动检测并部署
```

### Step 4.5 — 代码审查（Code Review）

> **为什么单独列出来？**
> Thor 不懂代码，无法做技术审查。这步的核心是：
> 1. Loke 强制"换位思考"——假设自己是另一个开发者来检查
> 2. 对照"教训文档"，避免重复踩坑
> 3. 自动化工具兜底，抓低级错误（花括号、语法等）

#### 4.5.1 自审清单（每次提交前必须过一遍）

| 检查项 | 说明 |
|--------|------|
| 变量命名 | 名字是否自解释？有无拼写错误？ |
| 边界情况 | 空值、异常输入是否处理？ |
| 硬编码 | 路径、ID、密钥是否提取为变量/常量？ |
| 控制台输出 | 有无遗留的 `console.log` / `print`？ |
| 缩进/格式 | 缩进是否一致？ |
| 括号匹配 | `{}`、`()`、`[]` 是否成对？ |
| 注释准确 | 注释描述和实际行为是否一致？ |

#### 4.5.2 教训文档对照

每次审查前，打开对应项目的 `docs/LESSONS.md`，检查：
- 过去犯过哪些错误？
- 本次改动是否涉及类似的代码模式？
- 如果涉及，必须额外验证

**教训文档路径**：`[项目]/docs/LESSONS.md`
（没有则创建，格式见本文档第八节）

#### 4.5.3 提交前自动化检查（推荐）

```bash
# Python 项目：运行语法检查
python -m py_compile main.py
python -m py_compile backend/*.py

# JavaScript/TypeScript 项目：
# npx eslint src/
# npx prettier --check src/

# 或一键检查（推荐配置 pre-commit hook）
```

> 💡 **教训记录（2026-04-08）**：
> - UI 改动在浏览器改完后，**必须 Ctrl+S 保存到文件**，否则重启后丢失
> - 解决：每完成一个独立改动，立即保存文件 + 做备份副本
> - 教训类型：内存 ↔ 文件同步疏忽

### Step 5 — 验收（UAT）
Thor 在 staging 环境测试，验证功能是否符合预期。

- ✅ 符合预期 → **书面确认**（哪怕一句"可以发布了"）
- ❌ 不符合预期 → 返回 Step 3，说明哪里需要改，记录变更原因

### Step 6 — 部署与监控
**合并到 main 并发布**：
```bash
git checkout main
git merge feature/功能名
git tag -a "vx.x" -m "功能描述"
git push origin main --tags
```

**发布后观察**：
- 观察 24-48 小时
- 检查功能是否正常、是否有报错
- 如有问题，立即回退到上一个 stable 标签

| 情况 | 是否记录 |

| 文档修改 | ❌ 不记录 |

## 五、回退操作

如果发布后发现问题，按以下顺序处理：

### 方案 A：Git 回退（推荐）
```bash
# 查看所有稳定标签
git tag -l "*stable"

# 切换到上一个稳定版本（不丢当前代码）
git checkout v1.0-stable

# 在此基础上新建修复分支
git checkout -b fix/紧急问题
```

### 方案 B：重新部署旧版本
```bash
# Render: 在 Dashboard → 找到上一个 deploy → 点击 "Create a rollback"
# Vercel:  deployments 列表中找到旧版本 → 点击 "..." → "Promote to Production"
```

## 八、知识产权保护（IP Protection）

> **核心原则**：每个产品从最初设计到每次迭代，都必须做好权利保护。
> **适用范围**：JobMatchAI 系列、BoligBeregner Danmark、未来所有产品

### 8.1 触发时机

以下节点必须完成 IP 检查：

| 节点 | 检查内容 |
|------|---------|
| 新项目启动 | 产品名、Logo、域名、商标查询 |
| 新功能发布 | 功能描述是否涉及第三方权益 |
| 品牌变更 | 商标、域名是否仍然有效 |
| 开源/公开代码 | 许可证、依赖库合规性 |
